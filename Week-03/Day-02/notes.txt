1. Killing Processes in Linux
------------------------------------------------------------------------
* Why Kill Processes?
    -> When a process consumes too many resources, behaves abnormally, or freezes, it is called a rogue process.
       Rogue processes waste resources and slow down important processes.
* Kill Command Basics: 
    -> Linux has 64 kill signals, but only a few are commonly used. If no signal is specified, the default is SIGTERM (15).

Command (Syntax: kill -signal PID):
kali > kill -signal PID

* Common Kill Signals:
    -> 1  (SIGHUP)    =  Stops the process and restarts with the same PID.
    -> 2  (SIGINT)    =  Interrupt signal, weak kill, usually works.
    -> 3  (SIGQUIT)   =  Creates a core dump, terminates process, saves info in core file.
    -> 15 (SIGTERM)   =  Default termination signal. Process stops gracefully.
    -> 9  (SIGKILL)   =  Absolute kill signal. Forcefully stops process, sends resources to /dev/null. 

Command (Restart process with HUP):
kali > kill -1 6996

Command (Force kill rogue process):
kali > kill -9 6996

Command (Kill by name without PID):
kali > killall -9 rogueprocess

* Using top to Kill:
    -> When top is running:
        Step-01  :  press K key.
        Step-02  :  Enter PID -> process will terminate.

⚠️ Rule to remember:
* Kill command = sends signal to terminate process. 
* SIGTERM      = default.
* SIGKILL      = absolute force kill.


2. Running Processes in the Background
------------------------------------------------------------------------
* Shell & Process Execution:
    -> In Linux, all commands are executed from the shell. When a command runs, the shell waits until the process finishes.
*Why Background Processes?
    -> Terminal gets blocked if a long process runs. Running in the background keeps the terminal free for other commands,
       allowing multitask saving of resources and screen space.

Command (Foreground Execution - Terminal stays busy):
kali > mousepad LinuxScript

Command (Run in Background via &):
kali > mousepad LinuxScript &
(Adding & runs the process in the background. Terminal immediately gives a new prompt for other commands).

Command (Move a running process to background via bg):
kali > bg <PID>
(If PID is unknown -> find it using the ps command).

* Practical Use:
    -> Long processes (scanner, exploit, editor, script) are better in the background.
    -> Hacker context : multiple terminals can run simultaneously.

⚠️ Rule to remember:
* Background process = keeps terminal free. Done with & or bg command.


3. Moving a Process to the Foreground
------------------------------------------------------------------------
* Foreground vs Background:
    -> Foreground process is active in the terminal and blocks the prompt.
    -> Background process leaves the terminal free while the process continues.
* Why Move to Foreground?
    -> Sometimes background processes need direct control, so they are brought to the foreground to attach the terminal back
       to that process.

Command (Syntax: fg <PID>):
kali > fg 1234
(Process 1234 moves to foreground. If PID is unknown -> use 'ps aux' to find it).

* Practical Use:
    -> Hacker context    : exploit/scanner moved back to foreground for control.
    -> Sysadmin context  : debugging critical services may require foreground.

⚠️ Rule to remember:
* fg = brings background process to foreground using PID.


4. Scheduling Processes in Linux
------------------------------------------------------------------------
* Why Scheduling?: System administrators schedule regular tasks like backups (e.g., Saturday 2 AM). Hackers schedule reconnaissance scripts regularly (open ports, vulnerabilities).
* Tools for Scheduling:
  - at command -> runs a task once in the future.
  - crond daemon -> schedules tasks regularly (daily, weekly, monthly).

Common Time Formats for at command:
* at 7:20pm          -> Runs today at 7:20 pm.
* at 7:20pm June 25  -> Runs on June 25 at 7:20 pm.
* at noon            -> Runs today at noon.
* at noon June 25    -> Runs on June 25 at noon.
* at tomorrow        -> Runs tomorrow.
* at now + 20 minutes-> Runs 20 minutes from now.
* at now + 10 hours  -> Runs 10 hours from now.
* at now + 5 days    -> Runs 5 days from now.
* at now + 3 weeks   -> Runs 3 weeks from now.
* at 7:20pm 06/25/2019 -> Runs on June 25, 2019 at 7:20 pm.

Interactive Mode & Command Example:
* Running 'at' shows the 'at>' prompt. Write commands there to execute at the scheduled time and finish with Ctrl+D.

kali > at 7:20am
at> /root/myscanningscript
(Here /root/myscanningscript will execute today at 7:20 AM).

⚠️ Rule to remember:
* at = schedules one-time future task, crond = schedules regular tasks.
